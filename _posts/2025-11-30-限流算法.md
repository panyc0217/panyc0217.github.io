## 固定窗口

统计一个时间窗口的请求计数，若计数超过阈值则限流，下个时间窗口计数清零

* 优点：实现简单，性能好
* 缺点：无法应对临界问题（窗口切换时可能允许双倍流量通过）
* 适用：限流精度要求不高的场景

基于String（key为固定窗口，val为窗口访问计数）：

```
// 1. 初始化固定窗口，设置过期以实现计数自动清零
exec("set 固定窗口 0 EX 过期时间 NX")

// 2. 固定窗口增加计数
result := exec("incr 固定窗口")

// 3. 若计数增加后未超过限流阈值，则允许流量通过
return result <= 阈值
```

基于Set（key为固定窗口，member为请求ID）：

```
// 1. 获取固定窗口原大小
c := exec("scard 固定窗口")

// 2. 固定窗口增加计数
exec("sadd 固定窗口 请求ID")

// 3. 若固定窗口为刚初始化状态，则设置过期以实现计数自动清零
if c == 0 {
  exec("expire 固定窗口 过期时间")
}

// 4. 若计数增加后未超过限流阈值，则允许流量通过
return c < 限流阈值
```

## 滑动窗口

统计多个时间窗口的请求计数，若计数总和超过阈值则限流，根据时间不断移除旧时间窗口和添加新时间窗口

* 优点：能够有效缓解固定窗口的“临界问题”，流量相对固定窗口平滑
* 缺点：针对突发流量的处理能力较弱（突发异常流量会使使窗口计数迅速到达阈值，导致后续正常流量被拒绝）
* 适用：限流精度要求相对高且不关心突发流量的场景

基于Sorted Set（member为请求ID，score为时间戳）：

```
// 1. 剔除过期计数
exec("zremrangebyscore 滑动窗口 0 窗口起始时间")

// 2. 获取当前滑动窗口计数
c := exec("zcard 滑动窗口")

// 3. 检查滑动窗口计数，若计数已经到达阈值则拒绝流量
if c >= 限流阈值 {
  return false
}

// 4. 若计数增加后未超过阈值，则允许流量通过
exec("zadd 滑动窗口 请求ID 当前时间")
return true
```

## 漏桶

请求时往桶里加入水滴（桶满时拒绝或阻塞），系统以恒定速率漏出水滴，需要等待到水滴漏出时才处理请求
* 优点：以恒定速率处理请求，流量平滑
* 缺点：无法应对突发流量，即使系统仍有余力，可能导致系统资源利用率低
* 适用：需要严格限流的场景（如数据库访问限流）

基于Hash（维护漏桶状态）：
```go
// ================================ Redis Server ================================

// 变量1：waters，当前水位，初始为0，下限为0，上限为capacity
// 变量2：capacity，水位上限
// 变量3：leakRate，漏水速率
// 变量4：lastTime，上次访问时间

// 1. 获取计算上下文
now := 传入的当前时间戳参数
waters, capacity, leakRate, lastTime := exec("hmget 漏桶 waters capacity leakRate lastTime")

// 2. 计算这段时间的漏水量
duration = now - lastTime
waters = max(0, waters - leakRate * duration)

// 3. 加入水滴，并更新water和lastTime
prevWaters := waters
nextWaters := prevWaters + 1
// 3.1 加入水滴失败，达到限流阈值
if nextWaters > capacity {
  exec("hmset 漏桶 waters ? lastTime ?", prevWaters, now)
  return false, 0
}
// 3.2 加入水滴成功，向Redis Client返回水滴漏出的等待时间，以保证漏桶的处理速率恒定
exec("hmset 漏桶 waters ? lastTime ?", nextWaters, now)
return true, prevWaters / leakRate

// ================================ Redis Client ================================

// 4. 等待水滴漏出
ok, duration := AcquireLeakBucket(time.Now())
// 4.1 返回限流报错
if !ok {
  return HTTP_CODE(429)
}
// 4.2 sleep等待水滴漏出，保证处理速率恒定
sleep(duration)
```


## 令牌桶

系统以恒定速率填充令牌到桶里，请求时从桶里获取令牌（桶空时拒绝或阻塞），获取令牌成功才处理请求
* 优点：与漏桶相较，流量同样平滑并且能够应对偶尔的突发流量
* 缺点：无法应对持续的突发流量
* 适用：系统平时稳定，偶尔需要应对突发流量

基于Hash（维护令牌桶状态）：
```
// 变量1：tokens，当前令牌数，初始为capacity，下限为0，上限为capacity
// 变量2：capacity，令牌数上限
// 变量3：fillRate，填充速率
// 变量4：lastTime，上次访问时间

// 1. 获取计算上下文
now := 传入的当前时间戳参数
tokens, capacity, fillRate, lastTime := exec("hmget 令牌桶 tokens capacity fillRate lastTime")

// 2. 计算这段时间的令牌填充数
duration = now - lastTime
tokens = min(capacity, tokens + fillRate * duration)

// 3. 获取令牌
nextTokens := tokens - 1
if nextTokens < 0 {
  exec("hmset 漏桶 tokens ? lastTime ?", tokens, now)
  return false
}
exec("hmset 漏桶 tokens ? lastTime ?", nextTokens, now)
return true
```
